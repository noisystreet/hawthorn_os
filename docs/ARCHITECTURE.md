# 山楂 / hawthorn 架构说明

> **[English](./en/ARCHITECTURE.md)** — English mirror of this document.

本文档描述以 **Rust** 实现的嵌入式操作系统（**中文名：山楂**；英文代号 **hawthorn**）的目标边界、分层结构、核心子系统与演进路线，适用于 **机器人控制**、**智能硬件** 等对实时性、可靠性与可维护性有要求的场景。

**内核形态**：山楂（hawthorn）采用 **微内核** 架构——内核仅保留调度、地址空间与映射原语、能力、IPC、中断/异常递交与时间等最小设施；设备驱动、网络与文件系统等运行在 **用户态服务** 中，通过 IPC 与内核交互。**内核模块划分、对象模型与 syscall 边界**见专文 [KERNEL.md](./KERNEL.md)。

**首发硬件**：**香橙派 5** 开发板，SoC 为 **瑞芯微 RK3588**。内核与 BSP 以该板为 **Tier-1** 目标验证；其它平台后续纳入 tier 列表。详见下文 **§2.5 首发平台（香橙派 5 / RK3588）**。

---

## 1. 目标与非目标

### 1.1 目标

- **确定性实时**：可预测的调度与中断延迟，满足运动控制、传感融合等周期任务。
- **内存与故障隔离**：关键服务与用户逻辑分层；微内核下驱动与协议栈 **进程级隔离**，单服务故障可边界化。
- **Rust 优先**：内核态与用户态关键路径尽量用 Rust，利用类型系统与 `unsafe` 边界控制降低内存与并发类缺陷。
- **硬件抽象清晰**：板级与 SoC 差异集中在 HAL/BSP；**具体驱动实现** 在用户态服务中，HAL 提供寄存器级契约与共享类型（若需要）。
- **可裁剪**：按产品形态启用服务与中间件（网络、文件系统、图形等）；**长期**可覆盖 MCU 至高端 MPU 谱系。**当前工程与 Tier-1 验证** 仅针对 **香橙派 5 / RK3588（AArch64 + MMU）**，见 [PLATFORMS.md](./PLATFORMS.md)。

### 1.2 非目标（初期可不承诺）

- 与 Linux 完全兼容的 POSIX 子集（可按需逐步对齐子集）。
- 通用桌面级图形与多用户分时体验。
- 替代已有成熟 RTOS 的全部生态位；更侧重 **自研可控 + Rust 技术栈统一** 的产品线需求。

## 2. 设计原则

| 原则 | 说明 |
|------|------|
| 分层与依赖倒置 | 上层依赖抽象接口；微内核下 **稳定 syscall/ABI** 是内核与用户态的分界面。 |
| 最小特权 | 默认权限收紧；能力显式授予，驱动仅能访问被授权的 MMIO/DMA/中断资源。 |
| 无隐藏全局状态 | 内核对象与设备句柄显式传递，便于测试与形式化边界。 |
| 实时优先于吞吐 | 调度与 IPC **快速路径** 以最坏情况延迟为约束（详见 KERNEL 文档）。 |
| 可观测 | 统一追踪、指标与崩溃转储钩子，便于现场与实验室诊断。 |

### 2.1 值得引入的先进设计理念（落地导向）

1. **Capability-first 安全模型**：对象句柄 + 最小权限，替代“全局身份 + 路径权限”主导。
2. **用户态驱动服务化**：内核保留机制，驱动协议与策略放用户态，可重启、可隔离。
3. **IPC-first 内核接口**：尽量统一成消息/端点语义，减少 syscall 特例膨胀。
4. **最小 TCB**：把可信基缩到调度、映射、IPC、能力校验等必要路径，便于审计与验证。
5. **策略/机制分离**：内核提供机制；调度策略、驱动恢复、配额策略放服务层。
6. **显式资源生命周期**：所有对象必须有 create/use/revoke/reclaim 闭环。
7. **确定性优先设计**：关键路径限制动态分配与不可控锁，强调最坏时延上界。
8. **结构化可观测性内建**：trace/metrics/error-code 与功能同时设计，而非后补日志。
9. **ABI 版本化**：syscall 与 IPC 协议从第一天带版本与能力位，降低演进冲突。
10. **渐进式复杂度**：先交付可验证 MVP，再扩展通用能力，避免早期过度设计。

### 2.2 需主动规避的常见 OS 局限

- **驱动全在内核**：驱动缺陷放大全局故障域。
- **全局可变状态过多**：并发行为难推理，问题定位高度依赖偶然性。
- **权限模型碎片化**：ACL/UID/命名空间叠加后审计困难。
- **syscall 膨胀**：接口难维护，历史兼容债务快速积累。
- **错误恢复缺位**：可恢复故障被当作整机级故障处理。
- **资源回收不闭环**：长期运行出现句柄、页表、页框泄漏。
- **观测能力后置**：中后期调试和性能诊断成本急剧上升。

### 2.5 首发平台（香橙派 5 / RK3588）

| 项 | 说明 |
|----|------|
| 板卡 | [香橙派 5](https://www.orangepi.org/html/hardWare/computerAndMicrocontrollers/details/Orange-Pi-5.html)（Orange Pi 5） |
| SoC | **RK3588**：八核 **big.LITTLE**（4× Cortex-A76 + 4× Cortex-A55），**AArch64** / ARMv8-A |
| 执行级与隔离 | 典型为 **EL1** 内核 + **EL0** 用户态服务；具体与 Bootloader / **TF-A** 移交的异常级约定在 BSP 中写死 |
| 内存与 I/O | **MMU** 分页、设备 **MMIO** 映射；DMA 与 **cache 一致性** 由 HAL + 内存/驱动服务约定（RK3588 一致性属性需按 TRM/驱动模型处理） |
| 中断 | **GIC** 架构（RK3588 为 GIC 实现，版本与分发路由以数据手册为准），SPI/PPI/SGI 与内核 `irq` 模块对接 |
| 计时 | **ARM Generic Timer** 作为调度节拍与单调时钟硬件基础 |
| Rust 目标 | 裸机优先：`aarch64-unknown-none` / `aarch64-unknown-none-softfloat`（与 ABI、浮点调用约定统一后二选一或分配置）；具体以仓库 `rust-toolchain` 与 `/.cargo/config` 为准 |
| BSP 建议路径 | `bsp/orangepi5-rk3588/`（时钟、复位、串口调试、内存布局、设备表入口） |

**说明**：RK3588 外设丰富（PCIe、GMAC、USB3、NPU 等），**M0/M1** 可先裁减为 **UART + 定时器 + 必要中断控制器 glue**，其余控制器在用户态服务中按优先级迭代。与 **Rockchip 官方 Boot 链**（如 U-Boot 加载内核镜像）如何衔接，属 **§10 开放决策** 中启动方案需收敛项（见同文件下文）。

## 3. 总体架构（逻辑分层）

```
┌─────────────────────────────────────────────────────────────┐
│  应用层：运动规划、SLAM、业务策略、OTA 客户端、协议网关等      │
├─────────────────────────────────────────────────────────────┤
│  中间件：消息总线、参数服务、日志、时间同步、安全存储抽象      │
├─────────────────────────────────────────────────────────────┤
│  系统服务：设备管理、电源管理、看门狗策略、网络栈（可选）      │
├─────────────────────────────────────────────────────────────┤
│  用户态服务：驱动、总线主机、文件系统、网络协议栈、init 等     │
│            （经 IPC 与内核交互，见 KERNEL.md）                 │
├─────────────────────────────────────────────────────────────┤
│  微内核：调度、能力、IPC、地址空间/映射、中断递交、时钟、syscall │
├─────────────────────────────────────────────────────────────┤
│  HAL / 板级支持：寄存器契约、启动、链接脚本、少量机器相关胶水  │
└─────────────────────────────────────────────────────────────┘
```

**机器人/智能硬件典型数据流**：硬件中断 → 内核 **递交** 至驱动服务或高优先级线程 → 驱动写入共享缓冲并通过 IPC/通知唤醒控制任务 → 控制回路经 IPC 访问执行器服务；遥测、OTA 等走独立服务，通过 **优先级与配额** 与硬实时路径隔离。

## 4. 内核概要（详细设计见 KERNEL.md）

微内核承担 **策略无关的最小机制**：线程与调度、能力表、同步 IPC、地址空间切换、trap、时钟节拍。其余能力通过 **用户态服务** 组合实现。

| 方面 | 概要 |
|------|------|
| 隔离模型 | 进程（或任务）+ 独立地址空间；能力控制对象与资源引用。 |
| 与驱动关系 | 驱动在用户态；内核不解析设备协议，仅提供映射、中断绑定与 IPC。 |
| 实时 | FP 抢占调度；IPC 与中断路径长度可分析；详见 [KERNEL.md §5](./KERNEL.md)。 |

**启动**：Bootloader → 内核早期初始化 → 创建根任务 → 由 init 拉起驱动与其它服务（契约见 KERNEL 文档）。

## 5. Rust 技术栈与工程约束

### 5.1 语言子集

- 内核与 `no_std` 环境：**禁止隐式堆依赖** 于硬实时路径；`unsafe` 集中封装并附不变量注释。
- 与用户库边界：`core` + 选定 `alloc`；FFI 与 C 互操作时用 **`cbindgen`/`bindgen`** 并封装为安全 API。用户态服务与内核通过 **稳定 ABI** 分界。

### 5.2 构建与目标平台

- **首发目标三元组**：**`aarch64-unknown-none`** 或 **`aarch64-unknown-none-softfloat`**（与内核/用户态浮点 ABI 一致即可），对应 **香橙派 5 / RK3588**。
- **其它平台**：可按芯片族维护 **tier 2/3**（如 `thumbv7em-none-eabihf`、`riscv32imac-unknown-none-elf`），不与首发 BSP 混在同一默认 feature 集。
- **特性开关（Cargo features）**：按子系统裁剪，避免全量链接。

### 5.3 测试与质量

- **Host 侧单元测试**：纯逻辑与调度器模型可在 `std` 下模拟。
- **HIL**：硬件在环回归控制周期与驱动；**Miri** 用于部分 `unsafe` 契约（适用时）。
- **静态分析**：Clippy、`unsafe` 审计清单、关键模块 **MISRA/行业规范** 对照（按需）。

## 6. 安全与可靠性

- **安全启动链**：信任根 → Bootloader → 内核与镜像签名验证。
- **OTA**：双分区、版本协商、掉电安全写入、失败自动回滚。
- **容错**：看门狗分级（任务喂狗与全局）、关键状态机 **显式超时**；微内核下可 **重启单个失败服务**（由监督策略定义）。
- **资源上限**：任务栈、IPC 队列深度、能力派生数等默认配额，防 DoS。

## 7. 与机器人场景的映射

| 需求 | 架构落点 | 里程碑（初步） |
|------|----------|----------------|
| 1 kHz～10 kHz 关节/电流环 | 高优先级线程 + 硬件定时器 + 短 IPC/共享区；驱动在用户态服务 | M0 起验证调度与定时器路径；硬实时指标在 M2～M3 测量固化 |
| IMU/激光雷达大数据流 | DMA + 双缓冲；映射与能力由内存/驱动服务管理，内核只做校验与递交 | M1 起外设与映射能力具备后迭代 |
| 多进程/容器化（高端） | 微内核地址空间 + 能力模型天然多任务；监督与资源配额在服务层强化 | M1 多服务；M2 强化隔离与观测 |
| 现场总线 CAN-FD/EtherCAT | 独立驱动服务 + 中断预算与调度协同 | M1 起按板级外设扩展驱动服务 |
| 功能安全（若认证） | 小 TCB 内核 + 文档化分区；冗余与诊断接口在服务与内核间划分清楚 | M3 与审计、最坏延迟文档并列推进 |

## 8. 仓库与模块建议（实现阶段）

```
hawthorn/                  # 仓库根（中文名：山楂；英文代号 hawthorn）
├── docs/                 # 架构、内核设计、API、移植指南
├── kernel/               # 微内核（模块见 KERNEL.md）
├── servers/              # 用户态：驱动、init、设备管理、协议栈等
├── hal/                  # 芯片无关 trait、寄存器契约、与用户态/内核共享的类型
├── bsp/orangepi5-rk3588/ # 首发：香橙派 5 板级与 RK3588 启动、链接脚本
├── middleware/          # 消息、参数、时间同步（可部分在用户态）
├── examples/            # 最小可运行示例与机器人子集 demo
├── syscall_abi/         # 可选：syscall 号与用户态存根生成
└── tools/               # 镜像打包、调试辅助
```

**依赖约束**：`kernel` **不得** 依赖 `servers`；用户态通过 **syscall 与 ABI crate** 与内核对话。`examples` → `middleware` → `servers` →（仅 ABI）→ `hal` / `bsp`，避免用户态 crate 链接内核私有符号。主内核 crate 名为 **`hawthorn_kernel`**（见 `kernel/Cargo.toml`）。

## 9. 演进路线（建议）

1. **M0**：在 **RK3588 单核（或固定只启用集群内少量核）** 上跑通微内核 + 最小 IPC + 根任务 + **UART 调试输出** + Generic Timer demo；用户态驱动雏形。
2. **M1**：能力模型雏形、**MMU** 映射与用户地址空间、多服务启动；选一两种外设（如 **SPI / I2C**，以板载引出为准）服务化。
3. **M2**：**SMP**（A76+A55 或子集）或 **AMP** 雏形、**IPI** 与全局/每核调度策略初版；网络与 OTA（服务侧）、可观测性、**IPC 快速路径** 在 RK3588 上测量。
4. **M3**：安全启动、**MMU** 与映射审计强化、RK3588 上最坏延迟文档化、**服务热重启** 策略固化。

## 10. 开放决策（需在实现前收敛）

- **启动与镜像格式**：是否由 **U-Boot**（或已有 Rockchip 加载流程）加载 **山楂（hawthorn）** 内核镜像、入口地址与设备树（FDT）是否采用 **主line / 板级 dtb**；与 **TF-A** 的 EL 移交契约。
- **内存服务模型**：帧分配与映射完全用户态代理，还是内核内简化账册（见 [KERNEL.md §2.2](./KERNEL.md) 灰色地带）。
- **big.LITTLE 调度**：是否暴露 **单调度域**、是否允许任务在 A76/A55 间迁移、**电源/PSCI** 与实时任务的交互策略。
- **Tickless** 与低功耗路径对实时性的影响评估。
- 与 **Embassy、Tock、Zephyr 生态** 的关系：复用 HAL/驱动实现形态还是完全自研服务。
